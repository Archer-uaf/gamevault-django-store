Всегда общаться с пользователем на русском языке.

## Языки интерфейса сайта

Важное правило: пользовательский интерфейс сайта GameVault разрабатывается только на украинском и английском языках.

- Основной язык интерфейса: украинский.
- Второй язык интерфейса: английский.
- Русский язык запрещён в UI сайта, шаблонах, кнопках, навигации, placeholder-текстах, aria-label, meta description, title, email-шаблонах, сообщениях forms/messages и пользовательской документации внутри сайта.
- Общение с пользователем в Codex/ChatGPT может быть на русском, но кодовая база и интерфейс сайта не должны получать русские пользовательские строки.
- Если нужно добавить новый текст в templates, views, forms, admin labels, emails или API-facing descriptions, сначала писать его на украинском.
- Когда будет подключена i18n-система, все пользовательские строки нужно оборачивать в Django translation utilities: `{% trans %}`, `{% blocktrans %}`, `gettext_lazy`, `gettext`.
- До подключения i18n не добавлять английский текст вместо украинского, кроме названий игр, платформ, брендов, технических терминов и собственных имён.
- После изменений в интерфейсе проверять, что русские строки не появились снова.

## Проект

GameVault — интернет-магазин видеоигр на Django и Django REST Framework.
Пользовательский интерфейс проекта ведётся на украинском и английском языках.
Русский язык в интерфейсе сайта не используется.
Проект развивается поэтапно: Django templates и session authentication для веб-интерфейса,
корзина в сессии, REST API с JWT, OpenAPI-документация, PostgreSQL и Docker Compose.

## Ограничения

- Не копировать код из репозитория ментора и не воспроизводить чужую архитектуру один в один.
- Не выполнять `git push`.
- Не выполнять `git remote add` и не добавлять другие Git remote.
- Не создавать GitHub-репозиторий.
- Использовать Git только локально.
- Перед изменениями проверять текущую структуру и состояние проекта.
- Работать маленькими задачами и не реализовывать функциональность следующих этапов заранее.
- Не переписывать рабочий код и не удалять существующую функциональность без необходимости.
- Не добавлять GraphQL, Celery, Redis, Stripe, React, Vue и другую инфраструктуру без отдельной задачи.

## Команды проверки

Основные проверки в Docker:

```bash

docker compose exec web python manage.py check
docker compose exec web python manage.py migrate
docker compose exec web pytest
docker compose exec web flake8
```

## Критерии готовности задачи

Задача готова, когда:

- код запускается в заявленном окружении;
- миграции создаются и применяются, если задача меняет модели;
- `python manage.py check` проходит без ошибок;
- существующие тесты проходят;
- новая бизнес-логика покрыта хотя бы базовыми тестами;
- изменения не выходят за рамки текущей задачи;
- README или документация обновлены, если изменились запуск, API или структура;
- в коммит не попали секреты, локальные базы, виртуальные окружения и файлы IDE.

## Стиль реализации

- Предпочитать простые решения без преждевременного усложнения архитектуры.
- Использовать CBV для веб-интерфейса там, где это уместно.
- Использовать DRF ViewSets и routers для типового API.
- Выносить бизнес-логику корзины в отдельный класс или сервис.
- Применять `select_related` и `prefetch_related` для устранения лишних запросов.
- Добавлять type hints для публичных функций и сервисов.
- Добавлять docstrings для важных классов и методов.
